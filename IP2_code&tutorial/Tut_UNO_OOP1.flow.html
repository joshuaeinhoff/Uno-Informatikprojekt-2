UNO: Einleitung für OOP1

Mit Hilfe dieses Tutorials entwickeln Sie Schritt für Schritt ein Spiel. Dabei werden nach und nach objektorientierte Eigenschaften eingesetzt. Außerdem können Sie das Spiel am Ende des Tutorials auf der Grafik-Konsole darstellen und spielen.

Spielkonzept

Bei dem Spiel, das Sie entwickeln werden, geht es um eine abgewandelte Version des Spiels UNO. Es handelt sich um eine Schatzsuche, wo der Spieler insgesamt vier Tempeln (Wasser-Tempel, Luft-Tempel, Erde-Tempel, Feuer-Tempel ) überqueren muss, um zu gewinnen und somit auch an den Schatz zu kommen. In jedem Tempel muss der Spieler Aufgaben lösen, um in den nächsten Tempel zu gelangen usw. Die zu bestehenden Aufgaben setzen sich aus einer Quizrunde und einem UNO-Spiel zusammen. Schafft der Spieler den Schatz zu finden, bekommt er als Belohnung 5Code Coins.

Die in jedem Tempel vom Spieler zu lösende Quizrunde besteht aus insgesamt 20 Fragen (5 Fragen pro Tempel) zu bestimmten OOP-Themen. Der Spieler hat x Sekunden Zeit die Frage zu beantworten. Wird die Frage richtig beantwortet, darf der Spieler weitergehen und UNO spielen. Ist dies jedoch nicht der Fall, bekommt der Spieler Minuspunkte oder muss sogar ganz von vorne anfangen, wenn alle Fragen falsch beantwortet wurden.

1 Karte

Implementieren Sie als erstes die abstrakte Klasse Karte, die alle gemeinsamen Eigenschaften aller Karten zusammenfasst.

1.2 Daten als Instanz-Variable implementieren

Eine Karte beschreiben Sie durch zwei Variablen:

  • farbe als Text 
  • nummer als Zahl

Definieren Sie die IV.

1.3 Superkonstruktor definieren

Damit Sie später Objekte der Klasse Karte erzeugen können, müssen Sie an dieser Stelle auch einen Konstruktor definieren. Der Konstruktor der Klasse Karte ist gleichzeitig der Superkonstruktor für alle Subklassen. Das heißt, dass wir für die Subklassen der Klasse Karte lediglich den Superkonstruktor aufrufen müssen, um Objekte erzeugen zu können.

Definieren Sie den Konstruktor der Klasse Karte, indem Sie die oben definierten IV initailisieren. 

1.4 Prüfen, ob eine Karte spielbar ist

Implementieren Sie die Funktion public boolean istSpielbar(Karte aktuelleKarte), die true zurückgibt, wenn eine Karte spielbar ist und false, wenn nicht.

Eine Karte ist spielbar, wenn:

  • ihre Nummer mit der Nummer der obersten Karte auf dem Ablagestapel übereinstimmt und/oder
  • ihre Farbe mit der Farbe der obersten Karte auf dem Ablagestapel übereinstimmt oder
  • die ausgewählte Karte eine Aktionskarte ist (d.h. die Farbe schwarz hat)

Vergleichen Sie die ausgewählte Karte mit der Karte auf dem Ablagestapel.

Hinweis

1.5 Abstrakte Methoden 

Deklarieren Sie:

  • die abstrakte Funktion istWelcheKarte(), die true zurückgibt, wenn die Kartenart übereinstimmt und false wenn nicht 
  • die abstrakte Prozedur erzeugeVisuelleKarte(), die die visuelle Darstellung einer Karte erzeugt und direkt danach versteckt
  • die abstrakte Prozedur setzeKarteAuf(), die die Karte auf eine gegebene Position setzt und sie sichtbar macht
  • die abstrakte Prozedur versteckeKarte(), die eine Karte versteckt
  • die abstrakte Funktion aktionDurchfuehren() zum Durchführen der Aktion einer bestimmten Karte
  • die abstrakte Funktion toString(), die uns die Karte als String zurückgibt

Diese Methoden werden Sie erst in den Subklassen von Karte implementieren.

3 Aktionskarten

Damit Sie UNO wie gewohnt spielen können, müssen Sie noch verschiedene Arten von Karten definieren. Diese haben einiges gemeinsam, nämlich die Eigenschaften, die sie von der Klasse Karte erben. Sie unterscheiden sich jedoch in der Darstellung und Reaktion auf einen Klick (wenn man beispielsweise eine Aktionskarte ablegt). Da die Aktionskarten von der Klasse Karte erben, müssen Sie diese als Subklassen dieser Klasse implementieren.

3.1 AussetzenKarte

Als erste Aktionskarte implementieren Sie die "Aussetzen-Karte". Diese hat folgende spezielle Eigenschaften:

  • Nachdem ein Spieler diese Karte abgelegt hat, wird der Gegenspieler „übersprungen“ und derjenige, der die Karte abgelegt hat, darf nochmal legen. 
  • Die Karte kann nur auf eine andere mit entsprechender Farbe oder eine andere AussetzenKarte gelegt werden.
  • Wenn diese Karte zu Beginn des Spiels gezogen wird, dann muss derjenige, der anfängt, aussetzen und der Gegenspieler darf beginnen.    

Implementieren Sie die Klasse AussetzenKarte als Subklasse von Karte.

3.1.1 Daten als IV implementieren

Definieren Sie folgene IV für die Darstellung einer AussetzenKarte:

  • die Kartenform als Rechteck
  • ein Kreis
  • eine Linie

3.1.2 Definition des Konstruktors

Damit Sie später Objekte von dieser Klasse erzeugen können, definieren Sie einen eigenen Konstruktor. Da die Subklassen von der Superklasse Karte erben, müssen Sie hier lediglich den Superkonstruktor aufrufen.

Für alle Aktionskarten, außer der Farbkarte, setzen Sie bei dem Aufruf des Superkonstruktors einen Wert für die nummer ein:

  • AussetzenKarte hat die nummer 10.

Grund dafür ist, dass wir auf dieser Weise einfacher mit den Aktionskarten umgehen können, da sie bei der Darstellung keine Nummer, sondern lediglich nur die Farbe schwarz haben. Sie können diese Nummer als eine ID betrachten.

Definieren Sie den Konstruktor der Klasse AussetzenKarte.

3.1.3 Kartenart zurückgeben

Implementieren Sie die Funktion public boolean istWelcheKarte(String kartenart), die die Kartenart der Karte zurückgibt. In diesem Fall "Aussetzen". 

Hinweis

3.1.4 Karte als String zurückgeben

Überschreiben Sie die toString()-Methode aus Karte, indem Sie die AussetzenKarte als String und die farbe zurückgeben.

  • Wenn diese Karte zu Beginn des Spiels gezogen wird, muss der Spieler, der anfangen darf, aussetzten und sein Gegenspieler darf anfangen.    

Implementieren Sie die Klasse RetourKarte als Subklasse von Karte.

3.2.2 Kartenart zurückgeben

3.2.3 Karte als String zurückgeben

  • Wenn sie zu Beginn des Spiels aufgedeckt wird, gelten dieselben Regeln.

3.4 WunschKarte

Diese Karte hat folgende Eigenschaften:

  • Der Spieler, der diese Karte legt, entscheidet welche Farbe als nächstes gelegt werden soll. Dabei darf auch die schon liegende Farbe gewählt werden (Für die Farbwahl wird es spezielle Farb-Buttons geben, die man drücken kann, um die gewünschte Farbe auszuwählen). 
  • WunschKarten kann man immer legen, das heißt man muss nicht auf die Farbe der vorherigen Karte achten.  
  • Wenn eine WunschKarte zu Beginn des Spiels gezogen wird, entscheidet der Spieler, der anfangen darf, welche Farbe als nächstes gelegt werden soll.   

Implementieren Sie die Klasse WunschKarte als Subklasse von Karte.
